# 游戏逻辑与数值文档

## 1. 能量与体力

### 能量（energy）

| 属性 | 值 | 可配置 |
|------|-----|--------|
| 最小值 | 0 | - |
| 最大值 | 100 | game_config.max_energy |
| 初始值（新用户） | 0 | - |
| 获取方式 | 占据节点后挖矿产出 | - |
| 消耗方式 | PK 失败/平局/拒绝/超时扣除 | - |

**规则**：
- 能量达到 100 时可发起 PK（`canPK = true`）
- 能量达到 100 时挖矿不再增加能量，且不消耗体力
- 每秒挖矿产出：5 点（`energy_per_second`）

### 体力（stamina）

| 属性 | 值 | 可配置 |
|------|-----|--------|
| 最小值 | 0 | - |
| 最大值 | 100 | game_config.max_stamina |
| 初始值（新用户） | 100 | game_config.initial_stamina |
| 消耗方式 | 挖矿时每秒消耗 1 点 | - |
| 恢复方式 | 每分钟恢复 1 点 | stamina_recovery_rate |

**规则**：
- 体力为 0 时无法继续挖矿（不产出能量）
- 能量达到 100 时挖矿不消耗体力
- 恢复为全局定时任务，每分钟执行一次

---

## 2. 挖矿机制

### 前置条件

1. 用户占据某房间内的一个节点（`occupy_node`）
2. 用户体力 > 0
3. 用户能量 < 100

### 产出规则

- **能量产出**：每秒 +5（`energy_per_second`）
- **体力消耗**：每秒 -1（仅当能量 < 100 时）
- **累计能量**：`total_energy` 累加每次产出的能量

### 定时任务

- 挖矿循环：每 1 秒执行一次
- 体力恢复：每 60 秒执行一次

## 3. PK 规则

### 发起条件

- 能量 >= 100（`canPK = true`，且见下）
- 已占据能量节点（当前房间内）；`canPK = energy >= 100 && 当前用户在该房间已占据节点`
- 目标为在线玩家或在线虚拟智能体
- 不能挑战自己

### 挑战流程

1. **发起挑战**：攻击者调用 `challenge_player`，向防御者发送 `pk_challenge` 事件
2. **挑战状态**：服务端在Redis中存储挑战状态（30秒TTL），用于超时检查
3. **响应方式**：
   - **接受**：防御者点击确认，进入PK界面，设置PK数值
   - **拒绝**：防御者点击取消，发送 `reject_pk_challenge` 事件，立即判输
   - **超时**：防御者30秒内未响应，自动判输

### 拒绝与超时判输

**触发条件**：
- 防御者明确拒绝PK挑战
- 防御者30秒内未响应（既不接受也不拒绝）

**判输规则**：
- 防御者（被挑战者）判输：能量 -50，losses +1
- 攻击者（挑战者）判赢：能量 +50，wins +1
- **不记录到 MySQL pk_records 表**（因为没有PK数值）
- 拒绝/超时**会写入 MongoDB battle_logs**，用于对战记录展示（见 docs/MONGODB_BATTLE_LOGS.md）
- 能量变化使用 `game_config` 配置：pk_energy_reward、pk_energy_loss

**超时保护机制**：
- 如果挑战者在30秒内离线，超时检查时不会判防御者输
- 只有挑战者在线时，超时才会触发判输

### 数值设置

- **King**：1-100，表示己方国王位置
- **Assassin**：1-100，表示己方刺客攻击目标
- 双方需在 `resolve_pk` 前通过 `pk_set_values` 设置，数值存 Redis 300 秒
- 拒绝/超时的情况不需要设置PK数值

### 胜负判定

**正常PK（双方都设置了数值）**：

**攻击距离**：
- 己方攻击距离 = `|己方 assassin - 对方 king|`
- 对方攻击距离 = `|对方 assassin - 己方 king|`

**判定**：
- 己方距离 < 对方距离 → 己方胜
- 己方距离 > 对方距离 → 己方负
- 相等 → 平局

**拒绝/超时PK**：
- 防御者直接判输
- 攻击者直接判赢

### 皮肤攻防对 PK 的影响（方案 A：距离修正）

双方设置 King/Assassin 后，先算**原始攻击距离**，再按当前佩戴皮肤的 `pk_attack`、`pk_defense` 计算**有效攻击距离**，比较有效距离判定胜负（有效距离更小的一方胜）。无皮肤或未设置 `current_skin_id` 时，攻防视为 0。

**原始距离（不变）**：
- 己方原始攻击距离 = `|己方 assassin − 对方 king|`
- 对方原始攻击距离 = `|对方 assassin − 己方 king|`

**攻击点数（pk_attack）**：
- 在己方原始攻击距离基础上减少己方有效攻击距离（己方更准）。
- 己方有效攻击距离 = 己方原始距离 **≥** 己方皮肤 pk_attack 时：己方原始距离 − 己方皮肤 pk_attack；否则不扣减，仍用己方原始距离。
- 边界：**若己方原始距离 < 己方皮肤攻击点数，则不使用皮肤攻击点数计算**（保持原始距离）。

**防御点数（pk_defense）**：
- 对方攻击己方时，增加对方的有效攻击距离（对方更难打中己方）。
- 对方有效攻击距离 = 对方原始距离 **≤** 配置上限时：对方原始距离 + 己方皮肤 pk_defense；否则不增加，仍用对方原始距离。
- 边界：**若对方原始距离 > 配置上限，则不计算己方皮肤防御属性**。配置上限由 `game_config.pk_skin_defense_distance_threshold` 控制（默认 30，范围 1–99）。

**胜负**：比较己方有效攻击距离与对方有效攻击距离，更小者胜；相等则平局。能量与战绩变化同前（胜 +50 / 败 -50 / 平 -50）。

**示例**：

| 场景 | 己方原始距离 | 己方 pk_attack | 对方原始距离 | 己方 pk_defense | 己方有效距离 | 对方有效距离 | 结果 |
|------|-------------|----------------|-------------|-----------------|-------------|-------------|------|
| 仅攻击生效 | 10 | 5 | 8 | 0 | 5 | 8 | 己方胜 |
| 攻击不生效（距离<点数） | 3 | 5 | 8 | 0 | 3 | 8 | 己方胜 |
| 仅防御生效 | 12 | 0 | 20 | 4 | 12 | 24 | 己方胜 |
| 防御不生效（距离>上限） | 15 | 0 | 35 | 6 | 15 | 35 | 对方胜 |
| 攻防同时生效 | 14 | 4 | 18 | 3 | 10 | 21 | 己方胜 |

### 能量变化

| 结果 | 攻击方能量 | 防御方能量 |
|------|------------|------------|
| 胜 | +50 | -50 |
| 败 | -50 | +50 |
| 平 | -50 | -50 |
| 拒绝/超时（防御者败） | +50 | -50 |

数值来自 `game_config`：pk_energy_reward、pk_energy_loss、pk_draw_energy_loss（默认均为 50）

### 平台池

- 平局时：平台池增加 `platform_pool_bonus`（默认 100）
- 平台池存在 `game_rooms.platform_pool`，用途可扩展（如活动奖励等）
- 拒绝/超时的情况不更新平台池

### 虚拟智能体

- 用户可挑战**在线**虚拟智能体（`virtual_ai_agents.status = 'online'`）；挑战时传入 `defenderId` 为虚拟智能体 ID。
- 匹配逻辑：若目标为虚拟智能体，服务端自动“接受”挑战，向挑战者发送 `pk_matched_virtual`（含 defenderId、defenderName），挑战者设置 King/Assassin 后调用 `resolve_pk` 与虚拟智能体数值结算。
- 胜负与能量变化与真人 PK 一致；记录写入 `pk_records`（attacker_type/defender_type 区分）及 `virtual_agent_battles`，并写入 MongoDB 对战日志。

## 4. 节点占据

- 每个房间 100 个节点（node_id 1-100）
- 每个用户在同一房间内只能占据 1 个节点
- 占据新节点时自动释放旧节点
- 节点被占用后其他用户无法占据，直到占用者释放或离开

### 退出登录与登录时的节点占据规则

**退出登录时**：
- 用户退出登录（Socket断开或调用HTTP登出接口）时，**自动释放用户占据的所有节点**
- 释放节点包括：更新数据库（`game_nodes.owner_id = NULL`）、清理Redis缓存、广播节点释放事件给房间内其他玩家
- 确保用户退出后，其占据的节点立即变为可用状态

**登录时**：
- 用户登录时，系统会**自动清理遗留的节点占据**（防止因异常断开导致的节点遗留）
- 如果检测到用户占据节点但未在线，会自动释放这些节点
- 确保用户每次登录时都处于干净状态，不会继承上次异常断开时的节点占据

**再次登录后的节点占据**：
- 用户退出登录后再次登录，如果需要占据节点，需要消耗设定的能量（`occupy_node_energy_cost`，默认50点）
- **首次免费占据标记**（`users.has_used_first_free_occupy`）不会被重置，确保每个用户只有一次免费占据机会
- 如果用户之前已使用过首次免费占据，再次登录后占据节点将消耗能量

## 5. 能量宝藏

管理员可在后台配置若干节点为「能量宝藏」点位，并设置每个点位的能量奖励金额。

**规则**：
- 用户通过随机降落或手动点击占据有宝藏的节点时，可获得配置的能量奖励
- **每个宝藏节点全局仅可被领取一次**（任何人先占先得），领取后该节点从 `game_config.energy_treasure` 中移除，管理员后台配置同步更新
- 能量奖励受 `max_energy` 上限影响，超出部分不累加
- 配置方式：管理端「游戏配置」→「能量宝藏配置」，选择节点并设置金额，保存即可

**配置格式**（`game_config.energy_treasure`）：JSON 数组，如 `[{"nodeId":5,"amount":30},{"nodeId":12,"amount":50}]`

### 5.1 智能宝藏

在固定宝藏之外，可开启「智能宝藏」：用户**切换能量节点**（非首次占据、已扣费）时，若该节点无固定宝藏或固定宝藏已被领取，则按配置的概率与比率进行判定，中奖时发放能量并写入 `treasure_claims`（`claim_type='smart'`）。

- **触发条件**：仅当本次为**非首次占据**（即切换节点且已扣除 `occupy_node_energy_cost`）时参与智能宝藏判定；若该节点已发放固定宝藏，则不再做智能宝藏判定。
- **概率与奖励**：在配置的 [prob_min, prob_max] 区间内取当前概率 p，以概率 p 随机判定是否中奖；若中奖，奖励 = 本次消耗 × (ratio / p)，再限制在 [amount_min, amount_max] 内。长期期望回报/消耗 = ratio。
- **与固定宝藏关系**：先判断该节点是否有未领取的固定宝藏并发放；若无，再在开启智能宝藏时进行概率判定。同一节点对同一用户仅可产生一条领取记录（固定或智能二选一，由 `treasure_claims.claim_type` 区分）。
- **配置与 AI 推荐**：管理端「游戏配置」→「智能宝藏配置」中设置开关、概率区间、比率、奖励区间；可使用「AI 推荐参数」调用当前 AI 提供商根据近期数据生成建议参数。

## 6. game_config 可配置项

| config_key | 默认值 | 说明 |
|------------|--------|------|
| energy_per_second | 5 | 每秒能量产出 |
| stamina_recovery_rate | 1 | 体力恢复速率（每分钟） |
| max_energy | 100 | 最大能量值 |
| max_stamina | 100 | 最大体力值 |
| pk_energy_reward | 50 | PK 胜利奖励能量 |
| pk_energy_loss | 50 | PK 失败损失能量 |
| pk_draw_energy_loss | 50 | PK 平局损失能量 |
| platform_pool_bonus | 100 | 平局时平台池增加能量 |
| energy_treasure | [] | 能量宝藏配置，JSON 数组格式 |
| smart_treasure_enabled | false | 是否开启智能宝藏（切换节点时概率中奖） |
| smart_treasure_prob_min | 0.05 | 智能宝藏中奖概率下限，0~1 |
| smart_treasure_prob_max | 0.25 | 智能宝藏中奖概率上限，0~1 |
| smart_treasure_ratio | 0.35 | 消耗与中奖能量总比率 |
| smart_treasure_amount_min | 10 | 智能宝藏单次奖励下限（能量） |
| smart_treasure_amount_max | 80 | 智能宝藏单次奖励上限（能量） |
| occupy_node_energy_cost | 50 | 玩家每次占据节点消耗的能量（首次占据免费） |
| pk_skin_defense_distance_threshold | 30 | 皮肤防御生效的对方原始攻击距离上限（≤此值时才加己方防御；>此值时不计算皮肤防御），1–99 |

管理端可通过 `PUT /api/admin/config` 修改以上配置。能量宝藏需在管理后台「游戏配置」中的「能量宝藏配置」区域进行可视化配置。

## 7. 剧情章节与任务

- **章节与任务**：剧情由 `story_chapters`（章节）与 `story_tasks`（任务）定义；用户进度存于 `user_story_progress`、`user_task_progress`。详见 [STORY_SYSTEM.md](STORY_SYSTEM.md) 与 [STORY_CONTENT.md](STORY_CONTENT.md)。
- **任务类型**：`chat_with_ai`（与 AI 智能体对话）、`occupy_node`（占据节点）、`mine_energy`（挖掘能量）、`find_treasure`（发现能量宝藏）、`reach_energy`（能量达到某值）、`complete_pk`（完成 PK 对战）等；每类任务可有 `target_value`（目标数值）。
- **进度驱动**：挖矿、占据节点、完成 PK、领取宝藏、与 AI 对话等行为由服务端或 Socket 检测后，可推送 `task_progress_ready` 或由客户端调用 `POST /api/story/tasks/:taskId/progress` 更新进度；任务达到目标后用户调用 `POST /api/story/tasks/:taskId/complete` 领取任务奖励（体力/能量）；章节内全部任务完成后可调用 `POST /api/story/chapters/:chapterId/complete` 领取章节奖励。
